# 🔍 EcoChain — Auditor Portal TODO
> Complete feature list for the Auditor Portal.
> Core principle: Auditor sees EVERYTHING — industry raw data AND how AI verified it —
> then signs digitally, which triggers report generation and blockchain storage.

---

## Auditor's Exact Responsibility in the System

```
Government assigns auditor to a submission
           │
           ▼
Auditor opens submission — sees TWO data layers:
  Layer 1: Raw industry data (what industry submitted)
  Layer 2: AI verification result (how the AI assessed it)
           │
           ▼
Auditor physically checks supporting documents:
  ✓ Fuel invoices
  ✓ Electricity bills
  ✓ Transport records
  ✓ Waste management certificates
  ✓ Production reports
           │
           ▼
Auditor makes decision:
  ✅ Approve     → continues to report generation
  ❌ Reject      → industry notified with reason
  ✏️ Request Correction → specific fields to fix
           │
           ▼
Auditor applies PKI Digital Signature
  (node-forge — signs the SHA-256 hash of the report data)
           │
           ▼
System auto-generates BRSR Report (PDFKit + AI narrative)
  Report contains:
  - Industry emission data (Scope 1/2/3)
  - AI verification result + risk score
  - Auditor decision + digital signature
  - Compliance notes per source
  - Baseline vs actual comparison
           │
           ▼
Report hash stored on Polygon blockchain (AuditRegistry.sol)
  Immutable record forever — tamper-proof
           │
           ▼
Report accessible to:
  ✅ Auditor (generated by them)
  ✅ Industry (their own report)
  ✅ Government (oversight)
  ✅ Public (hash verification only — no personal data)
```

- [ ] **PKI Signing Backend Logic** (`server/services/signature.service.js`):
  ```js
  // 1. Build report data object (all submission + decision fields)
  const reportData = {
    submissionId, industryId, period,
    totalCO2e, scope1, scope2, scope3,
    decision, remarks, complianceNotes,
    documentChecklist, auditorId,
    timestamp: new Date().toISOString()
  };

  // 2. Generate SHA-256 hash of the report data
  const reportHash = crypto
    .createHash('sha256')
    .update(JSON.stringify(reportData))
    .digest('hex');

  // 3. Sign the hash with auditor's private key (node-forge)
  const privateKey = forge.pki.privateKeyFromPem(auditorPrivateKeyPem);
  const md = forge.md.sha256.create();
  md.update(reportHash, 'utf8');
  const signature = forge.util.encode64(privateKey.sign(md));

  // 4. Get certificate hash (SHA-256 of auditor's public cert)
  const certHash = crypto
    .createHash('sha256')
    .update(auditorCertificatePem)
    .digest('hex');

  return { reportHash, digitalSignature: signature, certificateHash: certHash };
  ```

- [ ] **Signature validation before accepting**:
  - Verify auditor's certificate is not expired (`certExpiryDate > Date.now()`)
  - Verify passphrase unlocks the private key (node-forge decrypt)
  - If cert expired: block submission, show error "Renew your certification to continue"

- [ ] **After Successful Signature — Submission Sequence**:

  **Step 1** — Save `audit_reports` document to MongoDB:
  ```js
  await AuditReport.create({
    submissionId, industryId,
    auditors: [{ auditorId, auditorUserId, role, decision, remarks,
                 digitalSignature, certificateHash, signedAt: new Date() }],
    documentChecklist, complianceNotes, reportType: 'BRSR',
    finalDecision: decision,  // (or 'pending' if dual and only 1 signed)
    blockchainStatus: 'pending'
  });
  ```

  **Step 2** — Generate BRSR PDF (PDFKit):
  ```js
  const pdfBuffer = await pdfService.generateBRSRReport({
    submission, anomalyReport, auditReport, industryProfile
  });
  // Upload to Cloudinary
  const { secure_url } = await cloudinary.uploader.upload(pdfBuffer, {
    folder: `ecochain/reports/audit/${auditReportId}`,
    resource_type: 'raw', format: 'pdf'
  });
  await AuditReport.findByIdAndUpdate(auditReportId, { reportPdfUrl: secure_url });
  ```

  **Step 3** — Store report hash on Polygon blockchain:
  ```js
  const tx = await auditRegistry.storeReport(
    industryId,
    '0x' + reportHash,
    ethers.hexlify(ethers.toUtf8Bytes(digitalSignature)),
    verificationStatus   // 'single_verified' or 'dual_verified'
  );
  await tx.wait();  // wait for 1 block confirmation (~2 seconds on Polygon)
  await AuditReport.findByIdAndUpdate(auditReportId, {
    blockchainTxHash: tx.hash,
    blockchainStatus: 'confirmed',
    onChainRecordHash: reportHash,
    submittedToChainAt: new Date()
  });
  ```

  **Step 4** — Update submission status:
  ```js
  if (finalDecision === 'approved') {
    await EmissionEntry.findByIdAndUpdate(submissionId, { status: 'approved' });
    // → triggers credit/penalty calculation
  } else if (finalDecision === 'rejected') {
    await EmissionEntry.findByIdAndUpdate(submissionId, { status: 'rejected' });
  } else if (finalDecision === 'correction_requested') {
    await EmissionEntry.findByIdAndUpdate(submissionId, { status: 'correction_requested' });
  ```

## ⚙️ Backend — Services

- [ ] **`signature.service.js`** — PKI signing using node-forge:
  - `generateReportHash(reportData)` → SHA-256 of JSON-stringified report object
  - `signWithPrivateKey(reportHash, privateKeyPem, passphrase)` → Base64 signature string
  - `verifyCertNotExpired(auditorId)` → check `auditors.certExpiryDate > Date.now()`
  - `getCertificateHash(publicCertPem)` → SHA-256 of auditor's public certificate

- [ ] **`pdf.service.js` → `generateBRSRReport()`** — PDFKit BRSR layout:
  - Cover page: company name, CIN, period, EcoChain logo
  - Section A: General disclosures (auto-filled from industry profile)
  - Section C Principle 6: Scope 1/2/3 breakdown table
  - AI Verification block: risk score, flag, anomaly summary
  - Auditor Verification block: decision, remarks, compliance notes per source
  - Digital Signature block: auditor name, cert hash, signature string, signed timestamp
  - Blockchain Record block: report hash, txHash, Polygonscan URL (added after chain confirmation)
  - For dual audits: both auditor signatures in the signature block

- [ ] **`blockchain.service.js` → `storeAuditReport()`**:
  - Called after PKI signature applied
  - Params: `industryId, reportHash, digitalSignature, verificationStatus`
  - Calls `AuditRegistry.sol.storeReport()` on Polygon
  - Waits for 1 block confirmation
  - Returns: `{ txHash, blockNumber, confirmedAt }`
  - On failure: sets `blockchainStatus: 'failed'`, logs error, queues for retry

- [ ] **Blockchain retry mechanism** for failed transactions:
  - `blockchainSync.job.js` (every 5 min) checks `blockchainStatus: 'failed'` records
  - Retries `storeAuditReport()` up to 3 times
  - After 3 failures: sets `blockchainStatus: 'permanently_failed'`, alerts Admin

- [ ] **`audit.service.js` → `finaliseAuditDecision()`** — post-signature sequence:
  - Save audit_reports → generate PDF → upload to Cloudinary → store on blockchain → update submission status → notify all parties
  - All steps wrapped in try/catch — if blockchain fails, report is still saved in MongoDB

- [ ] **`audit.service.js` → `handleDualAuditProgression()`**:
  - After first auditor signs: update status to `awaiting_second_auditor`, notify second auditor
  - After second auditor signs: compute `finalDecision` (strict rules: any rejection = rejected), call `finaliseAuditDecision()`

- [ ] **RBAC middleware check for auditor data access**:
  - Auditor can ONLY access a submission if their `auditorId` is in `submission.assignedAuditors[]`
  - Build `verifyAuditorAssignment(auditorId, submissionId)` middleware
  - Apply to: `GET /api/emissions/:submissionId`, `GET /api/ai/results/:submissionId`, `POST /api/audit/verify`

- [ ] **`notification.service.js`** — auditor-specific notifications:
  - `audit_assigned` — sent when government assigns submission
  - `audit_second_signature_required` — sent to second auditor when first signs
  - `submission_resubmitted` — sent when industry corrects and resubmits
  - `deadline_approaching` — 3 days before submission deadline
  - `cert_expiry_warning` — 30 days before cert expires

- [ ] **`email.service.js`** — auditor email templates:
  - `audit_assigned.html` — "You have been assigned to audit [Industry] for [Period]"
  - `awaiting_second_auditor.html` — "Auditor 1 has reviewed [Industry]. Your signature is needed."
  - `deadline_reminder.html` — "Audit deadline in [X] days: [Industry] [Period]"

## 🔒 Security

- [ ] **Auditor can ONLY see their assigned submissions** — RBAC check on every data endpoint
- [ ] **Digital signature is non-repudiable** — once submitted cannot be changed; hash + signature stored in MongoDB AND on blockchain
- [ ] **Report hash covers all critical fields** — decision, emissions data, auditor ID, timestamp — any tampering breaks the hash
- [ ] **Certificate expiry check before signing** — expired cert = no signing allowed
- [ ] **Dual audit co-auditor cannot see each other's decision before both sign** — second auditor sees first decision only after government assigns them, not before (prevent anchoring bias). Actually, per spec, second auditor DOES see first auditor's decision — this is intentional for collaborative review
- [ ] **Rate limiting on `POST /api/audit/verify`** — max 10 submissions per auditor per hour (prevent accidental bulk submits)
